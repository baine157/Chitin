# Z420 Security Baseline

Date: 2026-04-24

Scope: read-only baseline for the Z420/OpenClaw host. This pass wrote only this packet and `docs/security/z420-security-agent.md`.

## Summary

No evidence found that this host installed the known-bad `@bitwarden/cli@2026.4.0` package. The installed Bitwarden CLI is `2026.3.0`.

Current posture decisions are tracked in
`docs/security/z420-security-posture-decisions-2026-04-30.md`.

The main current risks are exposure and hygiene, not confirmed compromise:

- SSH listens on all IPv4/IPv6 interfaces.
- Tailscale Serve exposes OpenClaw Control UI and a local service to the tailnet.
- Two OpenClaw gateway listener sets are present on loopback.
- A Chrome/OpenClaw browser profile is running with CDP on loopback.
- `BW_SESSION` appears in local logs/snapshots and should be redacted or quarantined.
- Chrome and OpenClaw updates are available.
- The deep OpenClaw security audit is blocked in the sandbox because it attempts to create plugin runtime dependency directories under `~/.openclaw`.

## Findings

### High

#### SSH listens on all interfaces

Evidence:

- `0.0.0.0:22`
- `[::]:22`

Risk:

SSH is reachable on any network interface unless blocked elsewhere. This may be fine if firewall/router policy is tight, but it should be an explicit decision.

Remediation requires approval:

- Confirm how the user connects to the machine.
- Verify firewall/router exposure.
- Consider restricting SSH to LAN/tailnet or key-only access.

#### Tailnet serves OpenClaw Control UI

Evidence:

- Tailscale Serve: `https://z420.<tailnet-domain>` routes `/control/openclaw` to `http://127.0.0.1:18789/control/openclaw`
- Tailscale Serve also routes `/` to `http://127.0.0.1:3001`

Risk:

This is tailnet-only, not public internet, but it exposes the Control UI to tailnet identity boundaries. That is acceptable only if the tailnet membership and ACL model are trusted.

Remediation requires approval:

- Review tailnet ACLs and shared devices.
- Decide whether Control UI should remain served over Tailscale.

### Medium

#### Duplicate OpenClaw loopback gateway listener sets

Evidence:

- `127.0.0.1:18789`, `127.0.0.1:18791`
- `127.0.0.1:19789`, `127.0.0.1:19791`

Risk:

Loopback-bound is good, but duplicate gateway listener sets can indicate stale processes or multiple active gateway contexts. That increases confusion during incident response.

Remediation requires approval:

- Identify which gateway is canonical.
- Stop stale duplicate gateway only after confirming active sessions are not using it.

#### Browser CDP listener is active on loopback

Evidence:

- Chrome listening on `127.0.0.1:18800`
- Browser profile path observed in process args: `~/.openclaw/browser/openclaw/user-data`

Risk:

Loopback-only CDP is much safer than public CDP, but it is still a sensitive local control channel for a signed-in browser profile.

Remediation requires approval:

- Close profile when credentialed browser work is complete.
- Consider periodic check for stale CDP listeners.

#### `BW_SESSION` persists in local logs/snapshots

Evidence:

Read-only file-name scan found `BW_SESSION` references in Codex/OpenClaw logs, shell snapshots, session JSONL, and memory summaries. Secret values were not copied into this packet.

Risk:

Even if expired, persisted session tokens are bad hygiene and train the system toward leaking local secret material.

Remediation requires approval:

- Targeted redaction in logs/snapshots.
- Quarantine original files only if needed.
- Deauthorize active Bitwarden sessions from the Bitwarden UI.

#### Updates available

Evidence:

- OpenClaw update available: `2026.4.23`
- apt upgrades available include `google-chrome-stable`, GTK packages, and `ubuntu-pro-client`

Risk:

Chrome update is especially relevant because browser automation and persisted profiles are part of this stack.

Remediation requires approval:

- Run `openclaw update`.
- Run apt security/package updates, or at minimum update Chrome.

### Low / Watch

#### OpenClaw security audit warnings

Evidence from `openclaw status`:

- Reverse proxy headers are not trusted.
- Potential multi-user setup detected.

Risk:

Not immediately critical if Control UI remains local-only and Telegram group boundaries are intentional, but these should be reviewed as the security agent design matures.

#### Firewall tooling not confirmed

Evidence:

- `ufw` not installed.
- `nft` not found.
- `firewall-cmd` not found.

Risk:

The baseline did not confirm an active host firewall. This matters because SSH is listening on all interfaces.

#### User crontab has one duty-hours job

Evidence:

- One Friday duty-hours heartbeat job is installed.

Risk:

Expected operational persistence, but security monitoring should include cron drift detection.

#### OpenClaw cron list failed once

Evidence:

- `openclaw cron list` returned gateway closed `1006 abnormal closure`.
- `openclaw status` and `openclaw health --json` otherwise reported the local gateway reachable/healthy.

Risk:

Could be transient, but should be watched because cron is part of the intended security-agent substrate.

## Blocked Checks

- `systemctl --user list-units`: failed with `Failed to connect to bus: Operation not permitted` in the sandbox.

These should be retried in a host-approved lane if we want a complete baseline.
