# Dreaming And Reflection Boundary

## Purpose

Dreaming is useful when it helps the Chief of Staff notice patterns that normal task execution misses.

Because `orchestrator/` is the principal-facing interaction surface, it may host bounded reflective synthesis. That does not make it a domain truth store and does not make reflection an execution proof.

## Where Dreaming Belongs

### `orchestrator/`

Allowed:

- CoS-level reflection on recurring friction, approval loops, runtime instability, and operator burden.
- Distillation of durable operating lessons into architecture docs, task packets, or explicit memory targets.
- Human-facing summaries that help Baine decide how the system should behave.

Not allowed:

- Treating reflection as proof that a task ran.
- Storing domain-specific truth that belongs in a lane.
- Mixing sensitive lane material into general reflections.

### `gbrain/`

Allowed:

- Source-grounded Research Fellow dream passes.
- Pattern recognition across vault notes, topic intake, audits, and project surfaces.
- Candidate destinations for later promotion into source, project, person, meeting, note, or ops artifacts.

Not allowed:

- Promoting dream output as canon without source links and readback.
- Absorbing onboarding, medical-log, portal, PHI, or case-log material unless explicitly sanitized and routed.

## Where Dreaming Does Not Belong

### `onboarding-ops/`

Credentialing and onboarding need evidence, canon promotion, validators, and hard approval gates. Dream diaries, autobiographical memory, and memory/dreaming runs increase ambiguity and should stay out.

### `medical-logs/`

Medical logging needs deterministic source-of-truth behavior, sanitized local state, and explicit append/prepare/submit gates. Dreaming creates the wrong affordance for PHI-adjacent work and should stay out.

## Rule

Dreaming can be powerful, but only when its output is treated as reflective signal.

No dream pass may replace:

- current source readback
- validator output
- command evidence
- portal readback
- approval gates
- domain-lane ownership
